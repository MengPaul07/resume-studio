# Structured Document Generation Protocol (SDGP) 想法记录

## 当前定位

- 这不是现在立刻开做的主项目。
- 目前只作为长期方向想法记录，用于后续继续思考、收敛和验证。
- 当前阶段目标不是实现，而是保留问题定义和抽象思路。

## 想解决的问题

- 许多文档生成任务并不是纯自由文本生成，而是带有明确结构、字段、约束和渲染目标的内容生成。
- 现有 LLM 工作流往往偏向：
  - 直接聊天生成整段文本
  - 解析已有文档并写回字段
  - 针对单一场景做 hardcode
- 缺少一个更通用的中间层，用来统一：
  - 结构化文档定义
  - 生成与编辑操作
  - 约束校验
  - 局部应用与版本演化
  - 渲染到不同目标格式

## 核心想法

- 设计一个面向结构化文档生成与编辑的协议/框架。
- 目标不是预先规定所有文档字段，而是：
  - 允许用户定义字段
  - 允许用户定义结构组织
  - 允许用户定义约束和渲染方式
- 框架负责提供通用运行时，而不是替所有 domain 写死 schema。

## 希望的形式

- 最理想的产物是一个 Python 库。
- 库的作用更像 protocol runtime / schema-driven framework。
- 不追求一开始就支持很多文档类型，而是先提供核心抽象和一个 reference domain。

## 可能覆盖的文档类型

- 简历
- 教案
- 试卷
- 法律文书
- 可能扩展到 PPT / slides

## 当前抽象方向

更值得统一的不是所有字段名字，而是这些元机制：

- schema 定义方式
- 节点/结构组织方式
- operation 语义
- constraint 表达方式
- proposal / apply / reject 工作流
- version / session 管理
- renderer 挂接方式

## 当前倾向

- 不强行做“所有文档共用同一套完整字段表”。
- 更倾向于：
  - 通用骨架
  - 用户自定义字段和结构
  - domain 自己补充语义

## 一个关键判断

- 这个方向更像“框架/协议设计”，而不是单个应用项目。
- 它更适合作为长期主线想法孵化，而不是短期冲成果项目。

## 暂时不做的事

- 不立即开新仓库重做
- 不现在就设计完整插件系统
- 不现在就支持多种文档类型
- 不现在就把它当作近期最重要任务

## 如果以后继续推进，第一步可能是什么

- 先写清楚一句话定义
- 先定义最小核心对象
- 先选一个 reference domain
- 先做最小 prototype，而不是大而全框架

## 一句话版本

> SDGP 目标是提供一套可扩展的、schema-driven 的结构化文档生成与编辑协议，使用户能够自定义字段、结构、约束和渲染方式，而框架统一处理生成、编辑、校验、应用和演化流程。
