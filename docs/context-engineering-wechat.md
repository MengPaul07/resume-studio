# 提示词之后：为什么你该开始做“上下文工程”

很多人做 AI 应用，卡在一个很常见的问题上：

同一个模型、同一套功能、甚至同一段提示词，今天聪明，明天失忆；这次能用，下次跑偏。

我们习惯把锅甩给模型能力，但在真实工程里，更多时候真正不稳定的，不是模型，而是你喂给模型的“上下文系统”。

这篇不讲某个项目怎么做炫技，而是把“上下文工程”拆成可执行的方法论：你可以直接拿去设计自己的 AI 产品。

---

## 一、先把概念说清楚：上下文工程不是“写更长的 Prompt”

很多团队把上下文理解成一段 system prompt，这太窄了。

更准确地说：

**上下文工程 = 围绕一次任务，把“模型做出正确决策所需的信息”按结构、时机、边界组织起来。**

这里至少有三层含义：

1. **结构化**：不是一坨文本，而是有字段、有类型、有约束。
2. **时序化**：什么先给，什么后给，失败后怎么补。
3. **可控化**：每次输出可追踪、可回退、可复用。

如果 prompt engineering 解决的是“这次怎么问”，那 context engineering 解决的是“系统长期怎么稳定地问对”。

---

## 二、上下文工程的 5 个核心面

### 1）任务上下文（Task Context）

回答三个问题：

- 你要模型完成什么目标？
- 成功标准是什么？
- 明确禁止什么？

常见错误是“优化一下这段内容”，这在工程上不可执行。  
你需要把它改成可验收表达，比如：

- “保持中文语气不变，只提升表达密度”
- “字数控制在 120~160”
- “不得新增未提供事实”

---

### 2）数据上下文（Data Context）

模型不能凭空推理产品状态。  
你必须明确“它看到的当前世界”：

- 当前文档对象（结构化 JSON）
- 本轮可编辑范围（scope）
- 历史版本（用于对比和回滚）
- 外部参考（如 JD、行业术语、知识库）

一句话：**没有状态，就没有可靠生成。**

---

### 3）工具上下文（Tool Context）

当模型可调用工具时，稳定性核心不在“能不能调”，而在“调完后是否可用”。

每个工具都应有明确契约：

- 输入字段（必填、可选、类型）
- 输出字段（machine-friendly）
- 失败语义（error code / reason）

比如在结构化写作场景里，`refine` 和 `suggest` 要严格分离：

- `refine`：返回**可直接应用**的候选修改（用于按钮应用）
- `suggest`：返回建议与理由（给用户理解，不直接落库）

---

### 4）记忆上下文（Memory Context）

对话历史不是越多越好。上下文窗口有限，噪声会吞掉关键信号。

实践上建议分层：

- **短期记忆**：最近几轮用户意图（保障连续对话）
- **长期记忆**：稳定事实（姓名、经历、偏好）
- **操作记忆**：最近 action 的摘要（做了什么、结果如何）

重点不是“全量保存”，而是“可检索地保存”。

---

### 5）控制上下文（Control Context）

当系统进入多步骤执行（解析、改写、建议、应用）后，需要一个控制层：

- 状态机或 DAG
- 节点事件流（开始、成功、失败）
- 可回滚版本点

你会得到两个好处：

1. 用户知道系统到底在做什么；
2. 出错时可以回到已知正确状态，而不是重来。

---

## 三、最常见的 4 个反模式

### 反模式 1：把所有信息塞进一段 Prompt

后果：冲突、冗余、不可维护。  
修正：把上下文拆为“任务层/数据层/工具层”，分别注入。

### 反模式 2：把聊天记录当记忆系统

后果：越聊越漂，响应越来越慢。  
修正：对 memory 做摘要化和结构化，只保留可复用事实。

### 反模式 3：工具输出是自然语言段落

后果：前端无法稳定渲染，按钮无法精确应用。  
修正：工具输出必须是 schema 化 JSON。

### 反模式 4：没有版本控制

后果：一次错误覆盖全部结果。  
修正：每个关键 action 产出版本节点，支持一键回滚。

## 四、一套可落地的最小实现（MVP）

如果你要从 0 到 1，我建议按这个顺序：

### Step 1：先定义“上下文协议”

至少定义这些字段：

- `task`：目标、约束、验收标准
- `state`：当前结构化文档
- `scope`：可编辑范围
- `memory`：短期摘要 + 长期事实
- `tools`：可用工具契约
- `version`：当前版本与父版本

### Step 2：再定义工具契约

每个工具至少统一：

- `input_schema`
- `output_schema`
- `status`
- `error`

### Step 3：最后做编排和可观测性

- 执行链路可视化（节点状态）
- 关键操作自动打版本点
- 支持 apply / reject / rollback

你会发现：模型没换，体验已经质变。

## 五、和项目的关系：只做一个轻案例

在我们的简历写作系统里，真正带来稳定性的，不是“某个更强模型”，而是三件事：

1. 把“建议”和“可应用候选”拆开；
2. 每次关键动作都生成可追踪版本；
3. 用节点状态告诉用户系统当前在做哪一步。

这说明一个事实：  
**上下文系统设计，往往比单点模型能力更决定产品体验。**

---

## 六、给团队的执行清单（可以直接贴到需求文档）

- [ ] 是否有统一上下文协议（不是散落字段）  
- [ ] 是否区分“可应用结果”和“解释建议”  
- [ ] 是否有 scope 机制，避免全量误改  
- [ ] 是否有版本点和回滚  
- [ ] 是否能看到每个执行节点状态  
- [ ] 是否把记忆压缩为结构化摘要  

只要这六项能落地，AI 应用的稳定性会明显提升。

---

## 结语

过去一年，大家都在卷模型参数和提示词技巧。  
下一阶段真正拉开差距的，是你有没有能力把“上下文”做成一个可维护、可演化、可复盘的系统工程。

一句话总结：

**Prompt 决定上限，上下文工程决定下限和稳定性。**
